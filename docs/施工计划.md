## 一、工作约定（先于一切代码）

### 1. 代码注释规范

- 注释用中文，写「为什么」不写「是什么」：约束、取舍、坑（如「方舟 endpoint 兼容 OpenAI SDK，但 base_url 必须带 /api/v3」），不复述代码字面行为。
- 每个模块头部一段中文 docstring：职责、在流水线中的位置、上下游数据格式。
- prompt 模板文件内用中文注释标明每段的设计意图（对应设计纪要的哪条决策）。

### 2. Git 工作流（大厂模式，双通道）

**核心区分：代码走 PR，数据产物由 bot 直推。**

| 通道 | 路径 | 流程 |
|---|---|---|
| 代码 | `src/`、`.github/`、`prompts/`、`eval/`、配置 | feature 分支 → PR → CI 必须绿 → squash merge 进 main，**禁止直推** |
| 数据产物 | `reports/`、`data/`、`assets/` | Actions bot 直接 commit main（ruleset bypass 名单放行 github-actions），commit message 带 `[bot]` 前缀 |

- 分支命名：`feat/<阶段号>-<简述>`（如 `feat/p2-arxiv-collector`）、`fix/`、`chore/`、`docs/`。
- Commit 遵循 Conventional Commits，描述可用中文：`feat(collector): 新增 arXiv 采集器与水位线过滤`。
- 每个施工阶段 = 1–2 个 PR，PR 描述写「对应设计纪要第 X 节 + 验收方式」。
- main 上配 ruleset：require PR + require CI（lint + test）通过；bypass 名单只加 Actions bot。
- CI（`ci.yml`，PR 触发）：ruff lint + pytest，与定时采集 workflow 分开。

### 3. 工具定义规范（OpenAI 模式）

方舟 DeepSeek 完全兼容 OpenAI SDK：客户端统一 `OpenAI(base_url="https://ark.cn-beijing.volces.com/api/v3", api_key=...)`。所有 function/tool 定义走标准格式，示例锁定规范：

```python
# 所有工具 schema 集中在 src/tools/schemas.py，一处定义、多处引用
# （LangGraph 节点内调用与未来 MCC/MCP 封装共用同一份 schema）
SCORE_ITEM_TOOL = {
    "type": "function",
    "function": {
        "name": "score_news_item",
        "description": "对一条 AI 资讯按'对 AI 应用工程的可用性'打分并归类板块",
        "parameters": {
            "type": "object",
            "properties": {
                "score": {"type": "integer", "minimum": 1, "maximum": 10,
                          "description": "重要性得分，评分依据 rubric 而非机构名气"},
                "category": {"type": "string",
                             "enum": ["model", "engineering", "paper"],
                             "description": "板块归属：模型动态/工程实践/论文新技术"},
                "reason": {"type": "string", "description": "一句话打分理由，中文"},
            },
            "required": ["score", "category", "reason"],
        },
    },
}
```

- 结构化输出一律走 tool call（而非让模型输出裸 JSON），返回后过 JSON Schema 校验（评测第 1 层）。
- 未来 MCP 化（阶段 9）时直接复用这些 schema，MCP tool 定义与 OpenAI function 定义字段基本同构。

## 二、施工阶段编排

> 排序原则：数据模型先于采集器（否则采集器没有落点）、采集器先于流水线、**观测先于复杂功能**（有 trace 再做实体关联和 judge，调试省一半力气）、推送和 MCP 最后（锦上添花不阻塞主链路）。

| 阶段 | 分支 | 内容 | 验收标准 |
|---|---|---|---|
| **P0 仓库与骨架** | `chore/p0-scaffold` | 建 GitHub 仓库（建议名 `ai-radar`）；目录骨架；uv + pyproject；ruff + pytest 配置；`ci.yml`；PR 模板；main ruleset（PR 必须 + CI 必须 + bot bypass）；README | CI 在空测试上跑绿；直推 main 被拒绝 |
| **P1 数据层** | `feat/p1-data-models` | Pydantic 模型：`NewsItem`（含 source_url、published_at、entities、category）、报告 frontmatter；`sources.yaml` 全 10 源配置；`state.json`（每源水位线）与去重历史的读写模块 | 单测覆盖水位线推进/失败不推进、去重判定 |
| **P2 采集器 ×4** | `feat/p2-collectors`（可拆 2 个 PR） | RSS / GitHub API / arXiv API / 网页抓取（Jina Reader 兜底）四个采集器；统一重试（3 次指数退避 5/15/45s，超时 30s）与失败原因分类；水位线过滤 | 用本地 fixture 数据单测；真跑一次能产出 NewsItem 列表 |
| **P3 LLM 层** | `feat/p3-llm-client` | 方舟 OpenAI 兼容客户端封装；`src/tools/schemas.py`（打分/实体抽取等工具定义）；第一阶段打分 prompt + rubric；tool call 结果的 schema 校验与重试 | 对 10 条 fixture 资讯打分，输出全部通过 schema 校验 |
| **P4 最小闭环** | `feat/p4-pipeline-v1` | LangGraph 流水线 v1：采集→去重→打分→深读（top3+中读3）→生成周中报 md+JSON→commit；`collect.yml`（cron 周二/五 UTC 0:00，先只留手动触发 workflow_dispatch）；报告尾注（缺失源/条目统计） | 手动触发 workflow，仓库里出现一份真实周中报，cite 全部可点 |
| **P5 观测与规则校验** | `feat/p5-observability` | LangSmith 环境变量接入；trace 标签（run_type/板块/源）；规则校验节点（cite 完整性/配额/字数）不合格重试；尾注加 token/成本/耗时实测 | LangSmith 后台能看到完整 trace；故意构造坏输出被校验拦截 |
| **P6 周报与历史关联** | `feat/p6-weekly-report` | 实体抽取入 `index.json`；周报流水线（读两份周中报 JSON：Top5+趋势段+一览+面试视角）；历史关联（实体匹配→旧摘要入上下文→演进关系）；`weekly.yml`（cron 周日） | 用两份真实周中报生成周报；人工检查至少一处历史关联正确 |
| **P7 推送与图片** | `feat/p7-delivery` | md→HTML→weasyprint PDF（装 fonts-noto-cjk）；SMTP 发信（正文 Top5 标题 + PDF 附件）；图片管线（模型挑 caption、程序下载入 `assets/`、md/PDF 嵌入） | 收到邮件：中文 PDF 无方块、图片正常显示 |
| **P8 评测深化** | `feat/p8-evaluation` | 忠实度 judge（摘要 vs 原文，低于阈值重生成/标低置信）；`eval/golden.jsonl` 骨架 + 回归脚本（改 prompt 前跑一致率） | judge 对人工构造的幻觉样例能拦截；回归脚本可运行 |
| **P9 MCP 化与 skills** | `feat/p9-mcp` | 挑 1–2 个采集 tool 封装为 MCP server（复用 P3 schema）；工作流 skills 文档化 | MCP server 可被客户端调用返回同等结果 |
| **P10 上线与校准** | `chore/p10-launch` | 打开两个 cron；试运行两周；按人工阅读体验校准 top-N、rubric、关键词，误判记入 golden set | 连续两周无人工干预产出 5 份报告 |

**里程碑**：P4 结束 = 系统「活了」（有真实报告产出）；P6 结束 = 设计的核心价值全部兑现；P8 结束 = 面试叙事完整。P0–P4 是严格串行的主链路，P5 之后 P7/P8/P9 之间无硬依赖，可按兴趣调整顺序，但 P5（观测）不要往后挪。

## 三、目录骨架（P0 交付）

```
ai-radar/
├── .github/
│   ├── workflows/          # ci.yml（PR 检查）、collect.yml、weekly.yml
│   └── PULL_REQUEST_TEMPLATE.md
├── src/
│   ├── collectors/         # 四个采集器 + 重试/水位线公共逻辑
│   ├── pipeline/           # LangGraph 节点（纯函数）与图定义
│   ├── tools/              # schemas.py：OpenAI 格式工具定义（单一事实来源）
│   ├── llm/                # 方舟客户端封装、prompt 加载
│   ├── report/             # md/JSON/PDF 渲染、邮件
│   └── validate/           # 规则校验、忠实度 judge
├── prompts/                # 打分/深读/judge 的 prompt 模板（带中文设计注释）
├── sources.yaml            # 10 源配置（加源 = 加一行）
├── data/                   # [bot 通道] state.json、index.json、去重历史
├── reports/                # [bot 通道] 按月分目录的报告
├── assets/                 # [bot 通道] 报告引用的图片
├── eval/                   # golden.jsonl、回归脚本
└── tests/                  # fixtures + 单测
```
