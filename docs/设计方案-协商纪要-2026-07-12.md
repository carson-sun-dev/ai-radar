# AI 前沿信息提醒 Agent — 设计协商纪要

> 日期：2026-07-12
> 目的：为国内大厂 AI 应用岗面试做准备，开发一个前沿信息收集与摘要 agent，同时借项目练手记忆管理、上下文工程、prompt 工程、MCP、function calling、skills/工作流编写。

---

## 一、初始计划

- 关注一线 AI 厂商（OpenAI、Anthropic、GLM、Qwen、豆包）的模型动态，以及应用层技术（LangGraph、LightRAG 等论文）。
- Agent 每 3 天收集一次信息并总结概括；每周做一次周总结，挑出最重要的几条，配中文摘要和技术流程介绍，节约阅读学习时间。
- 信息来源：官网、Ollama、论文等。
- 部署在自己的 2c2G 服务器上（已跑一个宣传网站，占用 800MB–1.2G，可升 2c4G），使用火山方舟的 DeepSeek v4 API。
- 报告 push 到专用 GitHub 仓库，按月份分文件夹存放。
- 历史报告逐步作为 RAG 挂到 agent 上，让新报告能关联旧技术的演进关系。

## 二、逐步协商：问题 → 解决方案

### 1. 信息源获取方式

**问题**：官网（尤其 OpenAI/Anthropic）有 Cloudflare 防护和 JS 渲染，直接爬取容易翻车；Ollama 无官方 API 且属于滞后的二手信息；国内厂商一手动态多在公众号，程序化获取困难且杂乱。

**解决**：优先结构化 API，少爬官网——arXiv API（论文）、GitHub API（org releases）、Hugging Face API（新模型 + Daily Papers）、官方 RSS；网页抓取只做兜底。国内厂商放弃公众号，统一按「GitHub org + 官网 news 页」获取。Ollama 排除出第一版。

### 2. 部署位置

**问题**：国内服务器的网络现实——`huggingface.co` 无法直连（需 hf-mirror 镜像）、GitHub API 间歇性抽风、OpenAI/Anthropic 官网直接屏蔽中国大陆 IP，采集环节放国内服务器必须配代理，稳定性和合规都是负担。另外内存虽然够（agent 峰值几百 MB），但常驻进程浪费。

**解决**：整个采集流程跑在 **GitHub Actions**（海外 runner）：免费（私有仓库 2000 分钟/月，实际每月用不到 150 分钟）、定时触发、能访问所有海外源、跑完直接 commit 报告到仓库。服务器完全退出本项目，留给未来的 AI 客服。已实测：火山方舟 API（`ark.cn-beijing.volces.com`）从美国网络访问返回 401（可达，仅差鉴权），延迟约 1 秒，GitHub Actions 上调用没有问题。

GitHub Actions 注意事项：

- cron 只认 UTC，且有 5–15 分钟触发延迟（无所谓）。
- 仓库 60 天无活动会禁用定时 workflow——本 workflow 每次 commit 报告即算活动，实际不会触发。
- runner 无状态：去重历史、实体索引等状态必须存在仓库里，跑完随报告一起 commit（仓库既是产出物又是数据库）。
- API key、SMTP 密码放 repo Secrets，以环境变量注入。

### 3. OpenAI / Anthropic 的信息获取

**问题**：两家是巨头不能忽略，除产品外的工程博客（应用问题分析与解决方案）对应用岗价值极高，但官网有防护。

**解决**（海外 runner 上问题大幅简化）：

- OpenAI：官方 RSS `openai.com/news/rss.xml`。
- Anthropic：无官方 RSS，用 RSSHub 公共实例或 Jina Reader（`r.jina.ai/<url>`）转 markdown 解析。**engineering blog（`anthropic.com/engineering`）单独列为一个源**——《Building effective agents》《Effective context engineering》这类文章就是应用岗面试题的出处。

### 4. 调度节奏

**问题**：「每 3 天采集 + 每周总结」不整除，会出现某周 2 次、某周 3 次采集的混乱。

**解决**：**周二、周五采集（UTC 0:00 = 北京时间早 8 点），周日出周报**。理由：美国公司发布集中在美西工作日（北京时间周三至周六凌晨），周五早采集正好收本周高峰，周日周报时最新信息只隔一天。周报**不重新采集**，只读两份周中报的结构化 JSON 做二次筛选合并——层级化摘要（hierarchical summarization），成本几乎为零。

### 5. Cite 溯源

**问题**：摘要必须附引用，确保信息真实、可溯源、可快速访问原文；但让模型生成 URL 是幻觉重灾区。

**解决**：**URL 由采集数据携带，不由模型生成**。每条信息从抓取起就带 `source_url` 和发布日期，一路跟随 JSON 到报告；prompt 明确「只允许总结提供的条目，每条输出必须附带给定的 URL」。

### 6. RAG / 历史报告关联

**问题**：想让新报告关联旧技术演进，但 2c2G 跑不动本地 embedding 模型，独立向量库（Milvus 等）对每周一份报告的语料量是杀鸡用牛刀；DeepSeek 没有 embedding API。

**解决**：现阶段**不部署向量库**，做轻量实体索引（与 LightRAG 双层检索思路一致）：

- 每份报告「一个文件两个受众」：YAML frontmatter（date、description、entities、tags）供机器读，正文 markdown 供人读——和 skill 的 description 机制同理。
- 维护总索引（`index.json` 或 SQLite）只存全部 frontmatter。
- 关联流程：新条目抽实体 → 查索引匹配 → 只把命中的旧条目**摘要**塞进上下文 → 模型写出「相比 X 月的 Y，这次改进在……」。

**未来迁移路径**（无锁定）：原始结构化 JSON 是本体，向量是可随时重建的衍生索引。当实体匹配明显漏检（语料过几百上千条）时，加 `sqlite-vec` 或 LanceDB（单文件嵌入式，GitHub Actions 里照跑），写一次性脚本批量补 embedding（一年语料成本几毛钱）。届时实体做精确召回、向量做语义召回，两路合并。

### 7. 重要性筛选（两阶段漏斗）

**问题**：全量深度摘要 token 成本高；且担心一阶段漏掉「标题朴素的重磅」（如 LightRAG: Simple and Fast RAG）。

**解决**：

- **第一阶段（便宜）**：用标题 + 摘要（绝不是只用标题）给所有条目打分，每条几百 token。
- **第二阶段（贵）**：只对 top 条目抓全文、清洗（trafilatura/readability）、深读，直接生成中文技术流程介绍（翻译是顺带产物，无独立翻译步骤）。省的大头是输入 token：20 条全深读 vs 只深读几条，差数倍成本。

防漏检四道保险：

1. 打分 rubric 写死在 prompt：按「对 AI 应用工程的可用性」打分（可复现性、是否解决生产真实问题、有无代码），明确「不要因机构名气或措辞低调而降分」。
2. 文本之外的信号进打分池：是否上 HF Daily Papers（人工精选）、GitHub star 增速、是否被订阅框架官方提及。
3. 低分不丢弃：全部进速览和实体索引，若某技术后续信号变强（被集成/再次出现），去重逻辑可触发重新打分——漏检可恢复。
4. 阈值宽容：深读之外设「中读」档，降低误杀代价。

### 8. 板块配额

**问题**：全局 top-N 有系统性偏差——模型发布类消息天然轰动，会持续挤掉对面试更有用的工程实践内容。

**解决**：报告固定三板块、各自独立排序、板块间不竞争：**模型动态 / 工程实践 / 论文·新技术**。工程内容席位保留，宁缺毋滥。

### 9. 去重

**问题**：同一条新闻会同时出现在 HF、官博、arXiv，不去重周报没法看。

**解决**：先 URL/标题哈希去重，再由 LLM 在汇总时做语义合并。去重历史存仓库（见第 2 点无状态问题）。

### 10. 提醒的最后一公里

**问题**：报告 push 到 GitHub 后仍需主动去看，不构成「提醒」。

**解决**：邮件推送周报（GitHub Actions 里 SMTP 发信很简单），收件人 [kaishuaisun0322@gmail.com](mailto:kaishuaisun0322@gmail.com)。**GitHub 存 markdown，邮件发 PDF 附件**：md → HTML → weasyprint 生成 PDF，CSS 控制样式；runner 需先 `apt-get install fonts-noto-cjk`，否则 PDF 中文是方块。邮件正文同时放 Top 5 标题列表（HTML），手机扫一眼标题即可决定是否打开附件。采集失败不发邮件（见第 11 点）。

### 11. 容错

**解决**：
- **重试策略**：每个源请求重试 3 次，指数退避（约 5s / 15s / 45s），单次超时 30 秒。三次全失败才判定源缺失。
- **失败不发邮件**，写进报告尾注：「本期 X 源缺失，原因：xxx」。原因分类记录——超时/连接失败（网络）、HTTP 403（疑似反爬，换 Jina Reader 兜底）、429（限流）、200 但解析失败（页面改版，需改代码）。单源失败不让整个任务失败。
- **任务级崩溃兜底**：整个 workflow 挂掉（连报告都没生成）时，GitHub Actions 默认给仓库 owner 发 workflow failed 邮件，零代码。源级失败走尾注、任务级崩溃走 GitHub 自带通知，互补。

### 12. 时效性窗口（水位线机制）

**问题**：采集必须限定时间范围（周二收上周五至周一、周五收周二至周四），否则新旧消息混杂，实体索引的时间线失真，历史关联失去意义。但硬编码日期窗口有隐患：某次采集失败，该窗口的消息永久漏掉。

**解决**：状态文件里给**每个源记水位线**（上次成功采集时该源最新条目的发布时间），每次只取「发布时间 > 水位线」的条目，成功后推进水位线。正常运行等效于固定窗口；某次失败后下次窗口自动拉长补上，不漏。**以条目发布时间为准，不是抓取时间**——RSS 有 `pubDate`、GitHub releases 有 `published_at`、arXiv 支持提交日期范围查询；网页抓取解析不到日期的，用首次见到时间 + 去重历史兜底。

### 13. 图/表截取

**问题**：周报中重要的架构图、流程图、数据表格，系统能否截取进报告？

**解决**：分三档，第一版做前两档：
- **表格**：HTML 表格转 markdown 表格，纯文本处理。
- **原文图片**（第一版就做）：博客和 arXiv HTML 版页面的图片是带 URL 和 caption 的 `<img>` 标签。深读时把图片列表（caption + 上下文）给模型挑 1–2 张关键图（架构图/流程图优先），程序下载存仓库 `assets/`，md 引用、PDF 嵌入。「模型挑、程序下」——模型不碰 URL，与 cite 同一纪律。
- **渲染截图 / PDF 内图片提取**（放二版）：需要 Playwright / PyMuPDF，复杂度上一个台阶；arXiv 基本都有 HTML 版，第二档已覆盖大部分需求。

### 14. 工作流框架

**问题**：LangGraph 之外是否有别的选择？有没有必要先纯手写再迁移？

**解决**：**直接用 LangGraph**（国内大厂 AI 应用岗 JD 出现率最高），不先手写——练 LangGraph 本来就是项目目的之一。唯一纪律：**每个节点写成纯函数**（输入 state、输出 state，不依赖框架特性），LangGraph 只负责编排、状态传递和重试，代码本身证明「业务逻辑与编排解耦」。Dify 不用于本项目，但要能在面试中说出它与代码框架的取舍；CrewAI/AutoGen（多 agent）本项目用不上。流水线：采集 → 去重 → 打分 → 深读 → 关联历史 → 落盘 → 推送。

### 15. 可观测性与评测

**问题**：国内大厂 AI 应用岗极其看重 Observability 和 Evaluation，原架构缺这一块；且需要能证明摘要不产生幻觉、成本可控。

**解决——观测（LangSmith）**：
- 选 LangSmith：与 LangGraph 原生集成，零代码，workflow 注入 `LANGSMITH_TRACING=true` + API key（进 Secrets）即全链路追踪。免费额度 5000 traces/月，实际用量不到十分之一。Langfuse（开源、可自托管）作为面试谈资：「有数据合规要求时我会自托管 Langfuse」。
- 每次运行一条 trace，LangGraph 节点自动成 span；记录每节点延迟、token 进出、成本；trace 打标签（`run_type: midweek/weekly`、板块、源）。
- 成本落地：每期报告尾注加「本期消耗 X tokens，成本 ¥Y，耗时 Z 分钟」——面试说的成本数字是每份报告上印着的实测值。

**解决——评测（四层，成本递增）**：
1. **规则校验**（零成本，每次跑）：cite 完整性（URL 必须来自本次采集集合）、打分输出过 JSON schema、板块配额与字数检查。不合格重试该节点。
2. **忠实度检查**（LLM-as-judge，每次跑）：每条深读摘要用独立 judge prompt 对照清洗后原文逐点核查技术断言有无依据；低于阈值重生成一次，仍不过则标「低置信 ⚠」。judge 调用同样进 trace。
3. **离线回归集**（改 prompt 时跑）：每周人工阅读时把误判/漏判记入 `eval/golden.jsonl`；改打分 prompt 前先回归看一致率——prompt 工程的单元测试，兼作 top-N 校准的数据来源。
4. **人工反馈闭环**：第 3 层数据来自每周 20 分钟阅读时顺手标注，零额外负担。

## 三、报告结构（定案）

### 周中报（周二/周五）


| 部分  | 条数              | 形态                                          |
| --- | --------------- | ------------------------------------------- |
| 深读  | 3 条（三板块各 1）     | 每条 300–500 字：是什么、解决什么问题、技术流程、对应用岗的意义，附 cite |
| 中读  | 3 条（三板块各 1）     | 每条 3–5 句 + cite                             |
| 速览  | 其余全部（约 10–25 条） | 一句话 + 链接，按板块分组                              |
| 尾注  | —               | 缺失/失败源标注、原始条目数统计、token 消耗/成本/耗时             |


### 周报（周日）


| 部分       | 内容                                                    |
| -------- | ----------------------------------------------------- |
| 本周 Top 5 | 板块保底各 1，剩 2 个名额全池竞争；复用周中深读稿重组，**增量是历史关联**（查实体索引写演进关系） |
| 趋势段      | 一段跨条目观察：本周信息拼起来说明什么方向（周报独有）                           |
| 本周一览     | 两次速览合并去重的全量列表                                         |
| 面试视角     | 1–2 个「本周可拿来在面试里讲的点」：一个技术 + 怎么讲的提示                     |


每周阅读负担约 20–30 分钟。数字跑一个月后按实际体验校准。

## 七、开发问题记录（施工期实战踩坑）

> 共性教训：这些问题都不是「代码写错」，而是**隐含假设在真实数据面前破产**——
> 平台 API 存在官方文档不写的行为；「新」这类日常词在系统里必须给出精确定义；
> 防偏差的规则本身也会制造新偏差；记账为「不阻塞」的技术债会在新出口重新爆发。
> 均为面试可讲的一手素材。

### 问题 1：方舟对长中文 tool call 参数约四成概率服务端丢弃（P4，2026-07-13）

**现象**：深读分析（300–500 字中文）走强制 tool call 时，3/3 全部失败；而短输出的批量打分 446 条几乎全部成功。失败响应的三个矛盾信号：

```json
{
  "choices": [{
    "finish_reason": "tool_calls",      ← 服务端宣称"工具调用完成"
    "message": {
      "content": "",                     ← 正文为空
      /* tool_calls 字段整个缺失 */       ← 工具调用也不见了
    }
  }],
  "usage": { "completion_tokens": 61 }   ← 输出 token 照常计费
}
```

**排查**：抓原始 HTTP 响应对比 SDK 解析结果——字段在 SDK 接手之前就已缺失，排除客户端嫌疑。5 次重复实验 2 次丢（约四成），且输出越长概率越高。

**根因**：tool call 的返回内容是「JSON 字符串值」，长中文分析全文需转义（引号→`\"`、换行→`\n`）；方舟服务端把模型原始输出解析成 `tool_calls` 结构这一步会间歇性失败并静默吞掉内容，文本越长越易触发。

**修复**（架构级，不是碰运气重试）：
- 长自由文本改走**普通 chat 补全**（内容直接放 `content`，零转义压力）
- 只有实体列表等**短结构化输出**继续走 tool call，重试提至 4 次（0.4⁴≈2.6% 残余失败率）
- 修复后 154/154 打分全成、深读零失败

**沉淀原则**：function calling 适用于「短、结构化」的输出；长文本生成塞进 tool arguments 是对通道的误用——「什么时候该用 function calling」的边界实证。

### 问题 2：时效无约束，首轮报告被存量旧闻霸榜（P4，2026-07-13）

**现象**：首份周中报的「模型动态」深读位被 `QwenLM/Qwen3`（仓库创建于 **2024-02-05**）占据——2026 年 7 月的「前沿」报告在深读两年前的旧闻。

**错误逻辑**（修复前 runner 只有一行过滤）：

```python
fresh = dedup.filter_new(items)   # 唯一的判定：这个 URL 我见过吗？
```

**根因**：「新」被定义为「我没见过」，它成立依赖两个隐含假设，双双破产：
1. 「采集器只返回最近一窗」——RSS 成立，但 GitHub org 采集器拉的是「最近创建的 50 个仓库」，跨度可达数年；
2. 首轮 `seen.json` 为空——一切都「没见过」，旧闻全部涌入，内容重磅照样 9 分霸榜。
   雪上加霜：当初为保住 arXiv 晚收录论文的补漏逻辑，特意不做客户端水位线过滤，等于拆掉了唯一的时间闸门。**「对我而言新」≠「新发生」**。

**修复**（时效闸，`max_age_days: 30`）：

```python
cutoff = datetime.now(UTC) - timedelta(days=30)
stale = [i for i in items if i.published_at < cutoff]
dedup.mark_seen(stale)      # 旧条目标"已见"：永不再进池，也不会每轮重抓重丢
fresh = dedup.filter_new([i for i in items if i.published_at >= cutoff])
```

收录条件从「没见过」升级为「**没见过 且 发布在 30 天内**」。30 天是权衡值：arXiv 晚收录通常晚几天，补漏保得住；存量旧仓库/旧博文全部挡住。效果实测：同批源 446 条 → 154 条，报告中 2024/2025 日期条目归零。

**遗留小项**（不阻塞，P10 校准）：① Anthropic 条目标题尾部粘着 Jina 链接文本里的日期字符串，需加标题清洗；② web 类源无真实发布日期（用首见时间），首轮会收进较老博文，运行两周后自然消失。

### 问题 3：头部模型旗舰换代被打低，深读位被时效 tie-break 错误决定（P4，2026-07-14 用户审查发现）

**现象**：首份周中报模型板块深读位是 Seed2.1（8 分），而 GPT-5.6 正式发布仅 7 分落在速览；且出现倒挂——GPT-5.6 Sol **预览**（8 分）比**正式发布**（7 分）分还高。用户审查一眼指出：OpenAI 旗舰换代在模型板块的重要性应高于 Seed 迭代。

**根因**（三层叠加，单独任何一层都不足以出错）：

1. **空洞文案 + 反光环规则 = 系统性打低**。打分只看标题+摘要（两阶段漏斗第一阶段），OpenAI 官方通稿是纯营销话术（"Frontier intelligence that scales with your ambition"），无任何具体能力信息；而 rubric 铁律①「不因标题低调降分」本意是防漏判小机构，副作用是把打分模型整体推向「内容判断优先」的保守侧——头部旗舰换代够不着「能力跃迁 9-10 档」的措辞。
2. **同一事件拆成三条，声量被稀释**：预览、正式发布、进驻 M365 各自独立打分，无事件聚合。
3. **同分 tie-break 放大偶然性**：选题层 8 分并列时新者优先，Seed2.1 恰好发布在采集当天；首轮 30 天存量窗口（问题 2 的修复值）让 6 月下旬的 GPT-5.6 条目与当天新闻同场竞争，进一步放大。

**修复**（rubric 校准，`prompts/score.md` 新增「头部机构模型发布的校准」段，与用户协商定案）：**不是**把头部发布一刀切进跃迁档，而是只给约 1 分上浮 + 按发布性质分档：

- 新一代旗舰正式发布 / 确立新范式（主版本号跃迁）→ 9-10
- 旗舰中期迭代 / 子线型号（x.5、Sonnet/Flash/Turbo 级）→ 7-8：值得了解升级点，不过分重视
- 预览版、进驻某产品、纯营销通稿 → 比照同型号正式发布低 1-2 分
- 并显式声明「不因官方通稿文案空洞降分」，按对应用层选型的实际影响判断

**效果实测**（同批 154 条离线重跑对照，`reports/2026-07/midweek-2026-07-13-v2.md`）：GPT-5.6 正式发布 7→9 分拿下深读位（理由：「主版本号跃迁，直接决定应用层选型方向」）；Sonnet 5 维持 8 分（符合「值得了解不过分重视」档）；Seed2.1 落回速览 8 分；分数分布从 6-8 聚团散开为 1-9 全区间（人事任命、政企合作类营销被正确压到 1-3 分）。

**残余偏差**（记账不掩盖）：① 预览版打了 9 分未按规则降 1-2 档——最终排序仍正确（正式发布日期更新胜出），但该判例应进 `eval/golden.jsonl`（P8）；② 本期深读+中读被同一事件（GPT-5.6 正式+预览）占据，是事件聚合缺失的第二个症状（第一个是声量稀释），聚合记 P10。

**沉淀原则**：反偏差规则本身会制造新偏差——「防大厂光环」防住了漏判小机构，代价是系统性低估头部发布，rubric 需要成对的校准规则对冲。更一般地：打分只看摘要时，「文案信息量」会冒充「内容重要性」，两者必须用显式分档规则解耦，不能指望模型从空洞通稿里自行推断重要性。

### 问题 4：首份周报门面三连坏——存量旧文占 Top5、脏标题、同文占两行（P6，2026-07-20 周报验收发现）

**现象**：首份周报（P6 上线后第一份产物）三处门面质量问题同时暴露：
1. **Top5 被 2024/2025 存量旧文占了三席**——「Effective context engineering（2025-09）」「Building effective agents（2024-12）」「Contextual Retrieval（2024-09）」全被标为「发布 2026-07-19」挤进本周最重要五件事；
2. **Top1 标题是脏的**：`FeaturedHow we contain Claude across productsAs agents grow more capable, so does…`——标签、标题、摘要粘成一行；
3. **同一论文占两行**：KV-Cache Grafting、SEED 各出现两次（arXiv 版与 HF papers 版）。

**根因**（三个独立缺陷在「全局 Top5 无日期防线」这一处汇合放大）：

1. **首见时间兜底 + 全局 Top5 = 存量洪峰直通门面**。问题 2 的遗留小项②早有记账：web 类源拿不到真实发布日期，用首见时间兜底。周中报有板块配额稀释影响，但周报的全局 Top5 无板块缓冲，存量旧文（内容确实好，9 分不冤）直接冲上门面。**「对我而言新」的老账在周报这个新出口上重新爆发**。
2. **标题清洗缺位**（问题 2 遗留小项①）：Jina 汇总节把卡片的 `Featured` 标签、标题、整段摘要连成一个链接文本，从未清洗。
3. **去重键绑定 URL**：`make_item_id` 用规范化 URL 做键，而同一 arXiv 论文有 `arxiv.org/abs/{id}v1` 与 `huggingface.co/papers/{id}` 两个入口，URL 不同 → 键不同 → 同文两条。

**修复**（全部落在采集/去重层治本，不在渲染层打补丁）：

- **日期即标题里那段脏字符**——一石二鸟：Jina 卡片标题尾部粘的 `Mon DD, YYYY` 正是唯一能拿到的真实发布日期。提取它 → 标题变干净，同时把真日期交给 runner 已有的 30 天时效闸，存量旧文自动出局（**零新增过滤逻辑，复用既有闸门**）。
- **图片 alt 作已知干净标题**：`Featured` 前缀直接剥；摘要粘连用页面图片 alt（`![Image N: 干净标题]`）作前缀匹配截回标题。
- **`dedup_key` 按 arXiv id 归一**：arxiv/HF 两入口收敛到 `arxiv:{id}`（去版本号），`make_item_id` 与周报 `week_pool` 共用；只改去重键，`NewsItem.url` 仍是原始可点链接。

**成果**（用修复后逻辑重处理已存 07-17/07-19 数据、仅 3 篇晋升条目补生成 ¥0.16，重出周报验证）：Top5 变为纯当周内容（how-we-contain 07-19、LongStraw 07-16、GPT-5.6 Sol、Gemini computer use、Seed2.1）；脏标题 0 处；同号论文去重检查为空。

**沉淀原则**：**遗留小项会在新出口重新爆发**——问题 2 当时判「运行两周自然消失」，但那是对周中报（有配额稀释）的判断；周报的全局 Top5 是个没有日期防线的新出口，同一个「无真实日期」缺陷在这里从「小项」升级成「门面事故」。记账为「不阻塞」的技术债，要在每个消费该数据的新特性上重新评估其影响面，而不是假设旧结论仍成立。另一条：**能从已有脏数据里反推的信息，就不必引入新数据源**——真实日期一直躲在脏标题里，提取它同时解决了标题和时效两个问题。
